# Backend Architecture

---

## 14.1 Tech Approach (Simple & Scalable)

Recommended stack:
- Backend: Node.js / FastAPI
- Database: PostgreSQL (primary storage)
- Cache Layer: Redis (IMPORTANT)

---

## 14.2 Why Redis is Needed (Your intuition is correct ✅)

Redis will be used for:

### 1. Real-Time Availability Cache
- Store room states in memory
- Fast reads for travel agencies

### 2. Instant UI Updates
- Pub/Sub system for live updates

### 3. Avoid Database Overload
- Frequent reads handled by Redis instead of DB

---

## 14.3 Data Storage Strategy

### PostgreSQL (Source of Truth)

Tables:
- hotels
- room_categories
- rooms

---

### Redis (Real-Time Layer)

Key Example:
hotel:{hotel_id}:category:{category_id}


Value:
```json
{
  "total": 7,
  "booked_rooms": [101,102,103],
  "available_rooms": [104,105,106,107]
}

14.4 Backend Flow (IMPORTANT)
When Hotel Books a Room:
API receives request → Book room 101
Validate:
Is room already booked?
Update PostgreSQL:
room.status = booked
Update Redis:
Add 101 to booked list
Remove from available list
Publish Event:
"room_updated" (Redis Pub/Sub or WebSocket)
Frontend updates instantly
14.5 Travel Agency Data Fetch

Flow:

Frontend → API → Redis (FAST)
If cache miss → fallback to PostgreSQL

14.6 Real-Time System
Option 1: WebSocket (Recommended)
Instant updates
Best UX
Option 2: Polling (Simple)
Every 5–10 seconds refresh
15. API Design (Basic)
Hotel Actions
POST /categories
POST /rooms
POST /book-room
POST /unbook-room
Travel Agency
GET /hotels
GET /hotel/{id}/rooms
16. Key System Decisions
Redis = speed layer (NOT source of truth)
PostgreSQL = final data storage
UI = grid-based interaction (core experience)
No complex booking logic initially
17. Final Architecture Summary

Hotel Action → API → PostgreSQL (write)
↓
Redis (update)
↓
WebSocket / PubSub
↓
Travel Agency UI (live update)

18. Product Strength (IMPORTANT FOR DEMO)

What makes this strong:

Real-world model (room-level tracking)
Instant visibility (no calls needed)
Scalable to full booking system later
Very simple UI → high adoption
